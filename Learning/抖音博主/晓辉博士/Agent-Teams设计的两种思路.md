# Agent Teams设计的两种思路：编程场景和研究场景

## 核心观点摘要

Multi-Agent协作在**确定性场景**（编程）和**不确定性场景**（研究）中需要完全不同的设计思路。

## 编程场景的 Multi-Agent（Anthropic 方案）

面向确定性任务，特征：

- **明确架构**：PM（项目经理）组织整体，下设执行 agent
- **明确验证机制**：对就是对，错就激励重做
- **agent 间少讨论**：避免噪声，执行就行
- **模型分层**：聪明模型（o1）做大脑/规划，快速模型（haiku）做执行
- 类似**工业时代**的流水线模式

典型案例：Anthropic 用此方式写 C 编译器

## 研究场景的 Multi-Agent（晓辉博士方案）

面向不确定性任务，关键区别：

### 共同点
- 通过文件交互（agent 把发现写到文件，下一个读取）
- 需要 PM 控制整体进度
- 需要审核机制

### 关键差异：4个额外角色

**1. 批判者（Critical Thinker）— 最关键**
- 站在第一性原理角度反思已有假设
- 挑战执行者的研究方向和方法
- 防止 agent 自圆其说
- 引入对抗性视角，产生完全不同的思考

**2. 探索者（Explorer）**
- 不在已有圈子里打转
- 主动寻找新视角、新文献、新方向
- 类似团队中预留20%探索时间的做法
- 创新的必要条件

**3. 专家间讨论**
- 不同学科的 agent 之间需要充分通信
- 通过讨论启发新思考
- 与编程场景"避免讨论"相反

**4. 总结者**
- 汇总各方向成果

### 研究场景的完整角色
1. PM（聪明大脑，规划决策）
2. 执行专家（不同方向，不同 prompt）
3. 批判者（挑战质疑）
4. 探索者（前沿延伸）
5. 总结者

## 金句

> 一般的 agent 会倾向于自圆其说，但有了批判的视角就有一种对抗。

> 编程场景特别像是工业时代，但面对不确定性问题，agent 之间需要讨论。

> 方向要去哪儿、为什么做、有什么价值——这些要人来补充。
