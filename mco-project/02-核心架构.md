## 核心组件

### 1. CLI 入口

MCO 提供两个主要命令：

| 命令 | 用途 |
|------|------|
| `mco review` | 结构化代码审查，输出标准化的 findings |
| `mco run` | 通用多 Agent 任务执行，不强制输出格式 |
| `mco doctor` | 环境健康检查 |
| `mco agent` | Agent 注册表管理 |

### 2. 任务调度器 (Orchestrator)

负责：
- 解析命令行参数
- 构建任务配置
- 并行调度多个 Provider
- 监控执行进度
- 处理超时与重试

### 3. 共识引擎 (Consensus Engine)

v0.9 版本的核心升级：

```
agreement_ratio = detected_by_count / total_providers_ran
consensus_score = agreement_ratio × max_confidence
consensus_level = confirmed | needs-verification | unverified
```

**共识等级含义**：

| 等级 | 条件 | 说明 |
|------|------|------|
| `confirmed` | ≥50% provider 报告 | 高置信度，多个 Agent 一致 |
| `needs-verification` | 2+ provider 但 <50% | 需要人工验证 |
| `unverified` | 仅 1 个 provider 报告 | 单一来源，可信度较低 |

## 适配器架构

### 统一适配器契约

每个 Provider 通过统一的适配器契约作为独立子进程运行：

```
┌────────────────────────────────────────────────────────┐
│                    Adapter Contract                     │
├────────────────────────────────────────────────────────┤
│  1. Detect  — 检测二进制文件和认证状态                    │
│  2. Run     — 启动 CLI 进程，传入提示词                   │
│  3. Poll    — 监控进程状态 + 输出字节增长，判断活跃度      │
│  4. Cancel  — stall timeout 或硬截止时 SIGTERM/SIGKILL  │
│  5. Normalize — 从原始输出中提取结构化 findings          │
└────────────────────────────────────────────────────────┘
```

### 扩展性

适配器架构可扩展 — 添加新的 Agent CLI 只需实现三个钩子：

1. **认证检查** (`auth check`)
2. **命令构建** (`command builder`)
3. **输出标准化** (`output normalizer`)

## 执行模型

### Wait-All 语义

```
执行模型是 wait-all：
- 单个 provider 超时或失败不会中断其他 provider
- 总耗时 ≈ 最慢的那个，而不是所有人的总和
```

### 进度驱动超时

```
Agent 自由跑完，仅在长时间无输出时取消
```

参数说明：
- `--stall-timeout`：无输出进展超过此时间才取消（默认 900 秒）
- `--review-hard-timeout`：硬截止时间（默认 1800 秒）

## 重试与容错

### 自动重试机制

| 场景 | 处理方式 |
|------|---------|
| 瞬态错误（超时、限流、网络抖动） | 自动重试，指数退避（默认重试 1 次） |
| 单个 provider 失败 | 不影响其他 provider |
| 结果缓存 | 每次调用都会真实执行，不复用缓存 |

## 运行模式

### Review 协作模式

| 模式 | 参数 | 行为 |
|------|------|------|
| **并行** | 默认 | 所有 provider 审查同一份范围 |
| **Chain** | `--chain` | 顺序执行，后一个 provider 看到前一个的分析 |
| **Debate** | `--debate` | 对合并后的 finding 再做一轮 challenge/refine |
| **文件分工** | `--divide files` | 按文件切片分工，大文件优先后轮转分配 |
| **维度分工** | `--divide dimensions` | 文件范围不变，但每个 provider 负责不同审查维度 |

> 注意：`--divide` 与 `--chain`、`--debate` 互斥

## 产物结构

当启用产物写入时，生成以下文件：

```
reports/review/<task_id>/
  ├── summary.md          # 人类可读摘要
  ├── decision.md         # PASS / FAIL / ESCALATE / PARTIAL
  ├── findings.json       # 聚合后的标准化 findings
  ├── run.json            # 机器可读执行元数据
  ├── providers/          # 各 provider 结果 JSON
  └── raw/                # 原始 stdout/stderr 日志
```

## 在 Claude Code 内运行

MCO 在启动 provider 子进程前会自动清理 `CLAUDECODE` 环境变量，可以安全地在 Claude Code 会话中运行 `mco`。
