# MCO 共识引擎

> 本文档详细说明 MCO 的共识引擎与去重机制

---

## 概述

MCO v0.9 将 review 合并从"简单去重"升级为"共识分析"。

核心公式：

```
agreement_ratio = detected_by_count / total_providers_ran
consensus_score = agreement_ratio × max_confidence
consensus_level = confirmed | needs-verification | unverified
```

---

## 共识评分详解

### Agreement Ratio（同意比率）

```
agreement_ratio = 检测到该问题的 Provider 数量 / 总共运行的 Provider 数量
```

示例：
- 3 个 Provider 运行
- 2 个 Provider 检测到同一个问题
- `agreement_ratio = 2/3 = 0.67`

### Consensus Score（共识分数）

```
consensus_score = agreement_ratio × max_confidence
```

示例：
- `agreement_ratio = 0.67`
- 各 Provider 的置信度：`[0.9, 0.8]`
- `max_confidence = 0.9`
- `consensus_score = 0.67 × 0.9 = 0.603`

### Consensus Level（共识等级）

| 等级 | 条件 | 说明 |
|------|------|------|
| `confirmed` | ≥50% provider 报告 | 高置信度，多个 Agent 一致 |
| `needs-verification` | 2+ provider 但 <50% | 需要人工验证 |
| `unverified` | 仅 1 个 provider 报告 | 单一来源，可信度较低 |

## 去重机制

### 跨 Agent 去重

MCO 收集每个 Agent 的发现，跨 Agent 自动去重相同问题。

#### 去重标准

两个 Finding 被视为相同，当：
1. 相同的文件路径
2. 相同的问题类型
3. 相似的描述（语义匹配）

#### 去重结果

去重后的 Finding 包含：

```json
{
  "id": "deduped-finding-1",
  "detected_by": ["claude", "codex", "gemini"],
  "agreement_ratio": 0.6,
  "consensus_score": 0.72,
  "consensus_level": "confirmed"
}
```

## 共识总结 (`--synthesize`)

### 功能

额外执行一轮 LLM 总结，输出跨 Agent 的共识/分歧摘要。

### 使用方法

```bash
mco review \
  --repo . \
  --prompt "Review for bugs." \
  --providers claude,codex,gemini \
  --synthesize \
  --synth-provider claude
```

### 输出内容

#### 1. 共识点

所有 Agent 一致认同的问题：

```
## Consensus

1. **SQL Injection in user_query.py** - All 3 providers agree this is a critical security issue.
2. **Missing error handling in API client** - 3/3 providers flagged this.
3. **Hardcoded credentials in config.py** - Confirmed by all providers.
```

#### 2. 分歧点

Agent 之间存在不同看法的地方：

```
## Divergence

1. **Performance in data_processor.py**
   - Claude: "This could be optimized with caching"
   - Codex: "No significant performance issue"
   - Gemini: "Suggests batch processing"

   Resolution: Consider Claude's suggestion, but prioritize based on actual profiling.

2. **Error handling approach**
   - Claude prefers explicit error types
   - Codex suggests using Result pattern
   - Gemini recommends exception wrapping

   Resolution: Team decision needed on error handling strategy.
```

#### 3. 建议行动

下一步应该做什么：

```
## Recommended Actions

1. **Immediate**: Fix SQL injection (HIGH priority)
2. **Short-term**: Remove hardcoded credentials
3. **Medium-term**: Improve error handling
4. **Discussion**: Decide on error handling strategy
```

## Chain 模式中的共识

### 功能

顺序执行 Provider，后一个 Provider 看到前一个的分析。

### 使用方法

```bash
mco run \
  --repo . \
  --prompt "Analyze architecture" \
  --providers claude,codex,gemini \
  --chain
```

### 执行流程

```
1. Claude 执行 → 输出 A
2. Codex 看到 A → 执行 → 输出 B
3. Gemini 看到 B → 执行 → 最终输出
```

### 适用场景

- 架构分析
- 复杂问题诊断
- 需要逐步深入的场景

## 辩论模式 (`--debate`)

### 功能

让多个 Agent 在最终输出前对合并后的 finding 再做一轮质疑或补充。

### 使用方法

```bash
mco review \
  --repo . \
  --prompt "审查这个 PR，质疑证据不足的 finding。" \
  --providers claude,codex,gemini \
  --debate
```

### 辩论流程

```
1. 各 Agent 独立审查 → Findings Set A
2. 合并 Findings → Findings Set B
3. 将 B 发回各 Agent 进行挑战/补充
   - 质疑证据不足的 finding
   - 补充遗漏的 finding
   - 确认共识
4. 最终输出经过辩论的 Findings
```

### 输出变化

辩论模式会：
- 降低 `unverified` finding 的权重
- 增加 `confirmed` finding 的置信度
- 标记被质疑的 finding

---

## 实践建议

### 1. 信任共识

- 优先处理 `confirmed` 级别的 findings
- `confirmed` findings 可以直接作为修复依据

### 2. 验证分歧

- 对 `needs-verification` 的 findings 进行人工复核
- 特别关注不同 Agent 有不同看法的地方

### 3. 谨慎对待单一来源

- `unverified` findings 需要额外验证
- 可能是 Agent 的误报或特殊情况

### 4. 使用 Synthesize

- 对于重要的审查，启用 `--synthesize`
- 获取更全面的共识分析

### 5. 结合辩论模式

- 对于高风险场景，使用 `--debate`
- 减少误报，提高审查质量
